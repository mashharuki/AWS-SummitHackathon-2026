問題1：タイムラインがすでに崩壊している

execution-plan.md のスケジュール（2026-05-09時点で策定）:

2026-05-10〜05-22: CDK実装 + shared/infra/agent のコード生成
2026-05-22〜05-30: api/web + デプロイ + デモ確認

現実（今日2026-05-16）:

---
問題2：AI-DLCのConstruction手順が予選には重すぎる

各Unitに対して以下のステージが必要です:

1. Functional Design
2. NFR Requirements
3. NFR Design
4. Infrastructure Design
5. Code Generation（Planning + Generation の2部）

7 Unit × 5 Stage = 35回のAI-DLCステージ実行

1ステージあたり平均30〜60分とすると、ドキュメント作業だけで17〜35時間。コードを書く時間が残りません。

書類審査では「ドキュメントの精緻さ」が評価軸でした。予選では「動くMVP」が評価軸です。AI-DLCプロセスの粒度を予選用に調整する必要があります。

---
リスク1（最重要）：Gmail API の本番利用に Google審査が必要

Gmail APIで他ユーザーのメールを読む場合、Googleの**OAuth同意画面の検証（Verification）**が必要です。

検証期間: 1〜4週間（場合によりそれ以上）
申請開始が必要だった時期: 4月中（今から申請しても予選に間に合わない可能性が高い）

execution-plan.md §7.1 の対応策:

▎ 「OAuth フローを先に疎通確認してから他の機能開発に入る」

これは対応策ではありません。Googleの審査をスキップする方法は存在しません（テストユーザー100人制限付きの「テスト」モードは使えますが、本番デモには不向き）。

現実的な判断: Gmail連携はMVPから外す。デモはSlack単体で成立させる。

---
リスク2：Slack Webhook受信は「公開URL」が先に必要

Slack Webhook（Events API）のエンドポイントURL検証は、SlackがそのURLにHTTPSでPOSTを送れる状態でないと完了しません。

手順:
1. AWS CDK で API Gateway を作成（U-02）
2. Lambda を作成しエンドポイントURLを取得（U-04）
3. SlackアプリのEvent Subscriptions にURLを登録
4. Slack からの検証リクエストに応答する実装が必要

つまり U-02（CDKデプロイ）とU-04（API実装）が完了しないとSlack連携のテストすら始められません。U-03a（エージェント実装）の動作確認が後回しになります。

execution-plan.md にこの依存関係が明示されていません。

---
リスク3：Bedrock AgentCore の ap-northeast-1 対応状況が未確認

execution-plan.md §7.2:

▎ 「AgentCore の代替として Bedrock InvokeModel 直接呼び出しを設計段階から用意しておく」

しかし component-methods/AG-01-task-extractor-agent.md や AG-02-sabori-proposer-agent.md
の実装は全てAgentCoreを前提に設計されています。「代替を用意する」とあるのに代替設計が存在しない状態です。

さらに深刻な問題：Claude Sonnet（Claude 3.5/4系）は ap-northeast-1 でネイティブに利用できるか確認済みですか？

一部のClaudeモデルはap-northeast-1未対応で、クロスリージョン推論（us-east-1等へのルーティング）が必要です。これはレイテンシに直接影響し、NFR-01「10秒以内」が
達成困難になります。

---
リスク4：Lambda Cold Start + Bedrock推論時間の合算

NFR-01a: Webhook受信 → タスク候補生成まで 10秒以内

実際のレイテンシ内訳（最悪ケース）:
Lambda Cold Start: 1〜3秒（Node.js + Hono）
Bedrock AgentCore 初期化: 1〜2秒
プロンプト送信 + 推論: 3〜8秒（Claude Sonnet、8,000トークン）
DynamoDB 書き込み: 0.1〜0.5秒

合計: 5〜13.5秒

10秒以内は最良ケースでしか達成できません。

AgentCore の多段呼び出しが入る場合（タスク整理AI含む）、20〜30秒超えも現実的です。NFRの値が実装の現実に即していません。

---
リスク5：SSEストリーミングはLambdaで追加設定が必要

application-design.md §7.2 のシーケンス図では API -->> FE: SSE delta 配信 として記載されています。

Lambda + API Gateway HTTP API でSSEを実現するには:
- Lambda関数URLまたはAPI Gateway のレスポンスストリーミングを明示的に有効化
- Transfer-Encoding: chunked の適切な設定
- Honoでのストリーミングレスポンス実装

これはよくある「書いたら動くと思っていたが実際は追加設定が必要」のケースです。execution-plan.md に記載なし。

---
リスク6：DynamoDB 8テーブル設計のアクセスパターン未定義

application-design.md でテーブル設計はありますが、GSI（グローバルセカンダリインデックス）の具体的なアクセスパターンが定義されていません。

DynamoDBはアクセスパターンを先に決めないと後で全テーブル再設計が必要になります（RDBと違いスキーマ変更がデータ移行を伴う場合がある）。これが実装中盤で発覚する
と大幅な手戻りになります。
