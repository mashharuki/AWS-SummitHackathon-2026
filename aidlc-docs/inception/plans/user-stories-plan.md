## 3. ストーリー粒度方針（Step 5）

### 採用アプローチ: **Epic-Based + User Journey ハイブリッド**

**選択理由**:
- 書類審査での可読性を最大化するため Epic 階層で構造化する
- 5分デモシナリオ（§5.2）との一貫性を保つため User Journey ベースでストーリーを流れに沿って並べる
- 審査員が「使い始めから価値享受まで」を一読で理解できる構成にする

### Epic 構成案（5 Epic）

| Epic ID | Epic 名 | 対応 FR |
|---------|---------|---------|
| E-01 | 外部ツール連携とタスク自動抽出 | FR-01, FR-07 |
| E-02 | タスク候補の管理（承認・編集・削除・手動追加）| FR-02, FR-08 |
| E-03 | 文脈を読んだサボり提案の受け取り | FR-03, FR-04, FR-06 |
| E-04 | 本音の記録（AIに蓄積させる）| FR-05 |
| E-05 | デモ・オンボーディング体験（将来ビジョン提示）| 参考付録 |

### INVEST 原則準拠方針

- **Independent**: 各ストーリーを Epic 内で独立して実装・テスト可能な粒度にする
- **Negotiable**: 受入基準を "Given-When-Then" で記述し、実装詳細は記載しない
- **Valuable**: ペルソナの「本音」に刺さる価値を Story の 'As a / I want / So that' に必ず反映
- **Estimable**: Story は 1〜3 日で実装できる粒度を目安にする（ハッカソン期間考慮）
- **Small**: 1 Epic = 2〜5 Story を目安にする
- **Testable**: 受入基準の Given-When-Then は、手動テストで検証できる具体性を持たせる

## 6. 事前検出された曖昧点（Step 4）

requirements.md と mockups を再分析して検出した潜在的曖昧点：

### 曖昧点 A: プライマリペルソナの解像度

**検出箇所**: requirements.md §1.2
- プライマリペルソナの「名前」「年齢」「具体的な1日のタスク数」「精神状態の起伏」が未定義
- デモシナリオでは「田中さん」と名前が出ているが、ペルソナシートとして未確定
- ストーリーの "As a / So that" の説得力がペルソナの具体度に直結する

### 曖昧点 B: MUST/SHOULD 境界の判断（FR-08）

**検出箇所**: requirements.md FR-08
- FR-08「手動タスク追加」は SHOULD だが、書類審査向けストーリーとして含めるべきか否かが不明
- mockups/01-task-list.png に + ボタン（フローティングアクションボタン）が確認できるため、デモ審査員に説明が必要になる可能性あり

### 曖昧点 C: エラー時のストーリー化範囲

**検出箇所**: requirements.md §5.3（エッジケース・エラーシナリオ）
- Slack OAuth エラー時・Bedrock タイムアウト時・DynamoDB 書き込みエラー時の挙動が技術要件として存在する
- これらをユーザーストーリーとして記述するか（例: 「連携エラー時にユーザーが通知を受け取る」）、受入基準に組み込むだけにするか、方針が未決

### 曖昧点 D: 将来展望ストーリーの記述範囲

**検出箇所**: requirements.md §1.3 MVP スコープ外
- 「取扱説明書」「複数人格」「SNS 機能」は MVP スコープ外
- 書類審査向けに「将来展望ストーリー（参考付録）」として E-05 に含めるか否かが未確定

### 曖昧点 E: 5分デモの「花形ストーリー」の優先表示

**検出箇所**: requirements.md §2.3 デモシナリオ核
- 審査員向けに 1〜2 件の「花形ストーリー」を先頭に持ってくるか、Epic 順で並べるかが未決
- ストーリー文書の読み手が「審査員」であることを考えると、デモシナリオとの紐付けを明示するか否かも未決
