## 1. プロジェクト概要

### 1.1 プロダクトの本質

SABOROU は、AI 時代に増え続けるタスクに対して「今どうサボれば一番うまく逃げ切れるか」をリアルタイムに提案する AI エージェントサービスである。

タスクを早く終わらせるためのツールではなく、Slack・Gmail・Google Calendar といった外部ツール上の文脈（締切・依頼者の温度感・リマインド状況・関連会議）を読み取り、「なぜ今サボれるのか・どこから危ないのか」を具体的な根拠付きで提案する。

提供価値は2層に分かれる：

| 層 | 内容 |
|----|------|
| **表向きの価値** | タスクに追われる人の心に余白を作る |
| **裏設定の価値（"人をダメにする"）** | タスク整理能力・優先順位判断・危機管理能力・締切感覚を AI に委ねることで、その能力が自然に退化していく |

### 1.1.1 ダメになる能力の具体例（裏設定）

| 能力 | 使用前（人間の状態） | 3ヶ月使用後（退化した状態） |
|------|-------------------|------------------------|
| **タスク整理能力** | メモ・メール・Slackを見て自分でタスクをリスト化 | AIが自動抽出するのを待つだけになり、メッセージを読んでも「何がタスクか」を自力で判断しづらくなる |
| **優先順位判断** | 締切・重要度・依頼者を考慮して優先順位を決定 | サボローの判定への依存が進み、「今やるべきか」を自力で決める機会が減る |
| **危機管理能力** | リマインドがなくても「そろそろ危ない」を察知できる | サボローの警告待ちになり、締切直前まで気づきにくくなる |
| **締切感覚** | 「あと2日」という時間感覚を体感で把握 | 「まだ大丈夫」の根拠を外部判断に依存し、内在化された時間感覚が弱まる |

### 1.1.2 サボり判定の科学的根拠（社会心理学・動機づけ理論）

SABOROU の「なぜ今サボれるか」という判定は、5つの確立された学術理論を設計基盤としている。
これにより、AIの恣意的な判断ではなく、人間行動の科学的メカニズムに基づく合理的提案であることを担保する。

| # | 理論 | 出典（査読済み） | 判定への活用 |
|---|------|----------------|-------------|
| 1 | **Collective Effort Model (CEM)** | Karau & Williams (1993). *Journal of Personality and Social Psychology, 65*(4), 681–706. | 「自分の貢献が価値ある結果につながる」と信じられないとき努力は合理的に低下する。締切が不明瞭・依頼者が認識していないタスクでは、今努力する価値が乏しいと判定 |
| 2 | **Identifiability（識別可能性）** | Williams, Harkins & Latané (1981). *Journal of Personality and Social Psychology, 40*(2), 303–311. | 個人の貢献が依頼者から「識別可能」な状態では社会的手抜きが消える。依頼者が offline / away でリマインドがない＝識別不能＝サボれる根拠 |
| 3 | **Sucker Effect（カモ効果）** | Kerr (1983). *Journal of Personality and Social Psychology, 45*(4), 819–828. | 依頼者自身が動いていないとき、「自分だけ先に動いて損をする」のを避ける心理が合理的に働く。依頼者の Slack ステータスや返信速度を参照 |
| 4 | **Self-Determination Theory (SDT)** | Ryan & Deci (2000). *American Psychologist, 55*(1), 68–78. | 外発的動機（締切プレッシャー・リマインド）がないとき、先延ばしが自律的に最適な戦略になる。リマインド回数・緊急度ラベルを定量化してLLMへ入力 |
| 5 | **Expectancy Theory** | Vroom (1964). *Work and Motivation.* Wiley. | 動機づけ＝期待（努力→成果）× 道具性（成果→報酬）× 誘意性。締切が遠い・フィードバックがないタスクは「今努力しても報われない」期待を生み、先延ばしが最適戦略となる |

**サボれる条件の論理式（5理論の統合）**:
```
[サボれる条件] = Identifiability が低い（依頼者が見ていない）
              ∧ Sucker Effect が発動（依頼者も動いていない）
              ∧ Expectancy が高い（締切まで余裕がある）
              ∧ SDT 外発的プレッシャーが低い（リマインドなし）
              → CEM によると集合的努力への動機が低下 = サボりが最適解
```

この科学的根拠に基づき、Claude Sonnet（Amazon Bedrock）がSlack・Gmail・Google Calendarから収集した文脈を評価し、「今サボれる理由」を人間が共感できる言語で説明する。

> 詳細な実装マッピングは [`application-design/component-methods.md`](../application-design/component-methods.md) §2「AG-02: SaboriProposerAgent」を参照。



### 1.2 ターゲットユーザーペルソナ

**プライマリペルソナ: 副業・フリーランサー**（Q16=B）

| 属性 | 内容 |
|------|------|
| **職業** | 副業エンジニア / フリーランスデザイナー / フリーランスライター |
| **状況** | 複数案件を掛け持ち、常にタスク過多の状態 |
| **ツール** | Slack（クライアントとのやり取り）/ Gmail（案件連絡）/ Google Calendar（締切・会議管理）を日常的に使用 |
| **ペイン** | 「何を今やるべきか」の判断で消耗する。どこまでやればいいかが分からない |
| **インサイト** | 「サボっていいよ」とは言われたい。しかしそれが根拠付きでないと信じられない |
| **心理** | 「もう少し寝かせていいかも」という本音を誰かに肯定してほしい |

**セカンダリペルソナ: ハッカソン審査員・テッキーな早期採用者**

技術的面白さとユーモア（「人をダメにする」コンセプト）で使い始め、実用性に気づく層。デモ演出のターゲットでもある。

### 1.3 MVP スコープ定義

```
MVP でやること:
1. 外部サービス（Slack/Gmail/Google Calendar）からタスクを拾い上げ、タスク化するAI
2. そのタスクに対して、文脈を読んでサボり方を提案するAI

MVP でやらないこと:
- ランキング / ゲーミフィケーション
- SNS 機能 / 共有機能
- 「自分の取扱説明書」生成（将来展望）
- 外部AIとの連携（ChatGPT / Claude 等）
- 複数人格切り替え（将来展望）
- Notion 連携（第4優先・MVP 外）
```

## 3. 機能要件（FR）

### FR-01: 外部サービス連携・タスク自動抽出

**優先度**: MUST  
**根拠**: Q1=E（Slack 第1優先 / Gmail 第2優先 / Google Calendar 第3優先）、Q2=C（3サービス同時連携）、Q4=B（Webhook駆動）

| 項目 | 詳細 |
|------|------|
| **連携サービス** | Slack（第1優先）/ Gmail（第2優先）/ Google Calendar（第3優先） |
| **トリガー方式** | Webhook / リアルタイムイベント駆動（メッセージ受信時に即時処理） |
| **実装SDK（Slack）** | Vercel Chat SDK（`chat` npm package）を採用し、Slack Event受信・署名検証・メッセージパースを型安全に実装 |
| **処理内容** | 外部メッセージを構造化タスク候補に変換（タスク名・締切・関係者・作業内容） |
| **エージェント** | エージェント①（タスク抽出エージェント）: Lambda + Bedrock AgentCore |

**受入基準**:
- Slack のメッセージ受信から10秒以内にタスク候補が生成される（Q10=B）
- タスク候補に「タスク名 / 締切 / 依頼者 / 作業内容」が構造化されて格納される
- Gmail の受信メール・Google Calendar の新規イベントからもタスク候補が生成される
- 生成されたタスク候補が DynamoDB「タスク候補」テーブルに書き込まれる

### FR-02: タスク候補の承認 / 編集 / 削除フロー

**優先度**: MUST  
**根拠**: Q3=E（モック準拠・3ボタン構成）、`mockups/01-task-list.png`（旧）／`ui/saborou_v2_02-tasklist.png`（最新UI・Pencilで起こした実装基準モック）

| 項目 | 詳細 |

- 承認済みタスクは「承認済みタスク」セクションへ移動する

### FR-03: 文脈読解・サボり提案の生成

**優先度**: MUST  
**根拠**: Q7=A（おっとり口調）、Q8=E（1行サマリ）、`mockups/02-task-detail-chat.png`（旧）／`ui/saborou_v2_03-detail.png`（最新UI）、`00-business-brief.md` §4

| 項目 | 詳細 |
|------|------|
| **エージェント** | エージェント②（サボり提案エージェント）: Lambda + Bedrock AgentCore |
| **入力** | 承認済みタスク + 周辺文脈（Slack温度感・Gmail・Googleカレンダー）|
| **出力1** | 「現在のサボり判定」（1行の簡潔な判定文）|
| **出力2** | 「判断材料」（箇条書きの根拠リスト）|
| **出力3** | おっとりサボローのチャットメッセージ（口調: 語尾「〜だよ」「〜かもぉ」・絵文字あり）|
| **人格（Persona）** | `saboru_ottori`（おっとりサボロー）固定。将来の複数人格化に備えて persona をデータとして外出し |

**表示仕様**（`ui/saborou_v2_03-detail.png` 最新UI / `mockups/02-task-detail-chat.png` 旧モック 参照）:
- 左ペイン「タスクの前提」ボックス: 締切・やること・依頼者を表示
- 左ペイン「現在のサボり判定」ボックス（淡黄色強調）: 1行判定文
- 左ペイン「判断材料」: 根拠の箇条書き
- 右ペイン「チャット」: サボローのおっとりメッセージ

**受入基準**:
- タスク詳細画面を開いてから10〜20秒以内にサボり提案が表示される（Q11=C）
- 提案文の語尾が「〜だよ」「〜かもぉ」等のおっとり口調で、絵文字が含まれる（Q7=A）
- 判断材料に外部ツール由来の根拠（リマインド有無・メール文言・会議日程）が含まれる
- 「今やるべき」状況では「危ない」と適切に警告する

## 4. 非機能要件（NFR）

### NFR-01: パフォーマンス — タスク抽出レイテンシ

**根拠**: Q10=B

- **目標**: 外部サービスからタスク候補を生成するまで **10秒以内**
- **計測開始**: Webhook 受信 / API 呼び出し起点
- **計測終了**: DynamoDB へのタスク候補書き込み完了
- **超過時のフォールバック**: CloudWatch Alarm + Lambda のタイムアウト設定（15分）で制御

---

### NFR-02: パフォーマンス — サボり提案レイテンシ

**根拠**: Q11=C

- **目標**: タスク詳細画面表示からサボり提案表示まで **10〜20秒以内**
- **ストリーミング表示**: Bedrock の推論中にテキストをストリーミング表示することで体感速度を向上させる
- **ローディング状態**: 推論中はスケルトン / ローディングアニメーションで待機状態を明示

---

### NFR-03: スケーラビリティ — 同時ユーザー数

**根拠**: Q9=A

- **想定同時ユーザー数**: 1〜5人（審査員・チームメンバーのみ）
- **Lambda 同時実行数**: デフォルト設定で十分（ハッカソン規模）
- **DynamoDB**: On-Demand モード（PAY_PER_REQUEST）で低負荷想定

### NFR-04: 可用性

**根拠**: Q14=D

- **SLO**: 特に定義しない。サーバーレスの自動スケールに任せる
- **依存**: Lambda / API Gateway / DynamoDB の AWS マネージドサービス可用性に準拠
- **ダウンタイム許容**: ハッカソン期間中はデモ時のみ確実に動作すればよい

---

### NFR-05: データ保持

**根拠**: Q12=C

| データ種別 | 保持方針 |
|-----------|---------|
| **外部ツール由来の生データ**（Slack本文・Gmail本文等） | 処理後即削除（DynamoDB には保存しない）|
| **タスク候補・承認済みタスク** | 永続保持（将来の分析・取扱説明書生成の原料）|
| **サボり提案ログ** | 永続保持（提案履歴・判断根拠の記録）|
| **本音データ**（クイック返信・自由入力） | 永続保持（将来の取扱説明書生成の最重要原料）|

### NFR-06: コスト管理

**根拠**: Q13=B、03-aws-architecture-policy.md §7

| 項目 | 目標値 |
|------|-------|
| **Bedrock 1リクエストあたりトークン数** | 8,000トークン以内 |
| **Bedrock 月額コスト** | $50 以内 |
| **AWS 全体月額コスト** | $10〜$70（目安）|
| **コスト管理** | AWS Budgets で $50 警告・$100 停止検討アラート設定 |

アプリ側でトークン数をガードし、超過しないようにプロンプトを設計する。

### NFR-07: セキュリティ・PII 保護

**根拠**: Q15=D（最小化・匿名化＋生データ不保持）、03-aws-architecture-policy.md §4

| 項目 | 方針 |
|------|------|
| **PII 最小化** | 依頼者名は仮名化 / イニシャル化を検討。本文は要約のみ保存 |
| **生データ不保持** | 外部ツール由来の本文（Slack/Gmail）は処理後即削除。DynamoDB にはサマリのみ保存 |
| **OAuth トークン** | AWS Secrets Manager に暗号化保存 |
| **API キー** | コードにハードコードしない。CI でも検知 |
| **HTTPS** | 全通信 HTTPS 必須（API Gateway / CloudFront） |
| **IAM** | 最小権限原則。CDK で明示的に権限を絞る |
| **暗号化（保存時）** | S3 / DynamoDB は AWS 管理 KMS で暗号化（デフォルト）|

**注**: Q23=B より Security Baseline Extension は不採用（PoC・プロトタイプ扱い）。ただし上記の基本セキュリティは実装する。

## 5. ユーザーシナリオ

### 5.1 主要ユーザージャーニー

#### ジャーニー1: 新規タスク候補の確認と承認

```
1. ユーザーが Slack でクライアントから「明日までに資料ざっくり作っておいて！」と受信
2. SABOROU の Webhook が Slack イベントを受信（Webhook トリガー）
3. エージェント①がメッセージを解析し、タスク候補を生成（10秒以内）
   → タスク名: 「提案資料の初稿作成」
   → 締切: 明日
   → 依頼者: クライアント（Slack DM）
4. タスク一覧画面の「承認待ちタスク」に新しいカードが表示
5. ユーザーが「承認する」ボタンをクリック
6. タスクが「承認済みタスク」セクションへ移動
7. エージェント②がサボり提案を生成開始
```

#### ジャーニー2: タスク詳細でサボり提案を確認

```
1. ユーザーが「承認済みタスク」のカードをクリック（タスク詳細画面に遷移）
2. 左ペインにタスクの前提情報（締切・やること・依頼者）が表示される
3. エージェント②がオンデマンドで文脈を再評価（10〜20秒以内）
4. 「現在のサボり判定」に1行の判定文が表示される
5. 「判断材料」に根拠の箇条書きが表示される
6. 右ペインのチャットでサボローが「まだ寝かせてOKだよぉ〜 ☁️」と提案
7. ユーザーがクイック返信「たしかに、まだ寝かせる」をクリック
8. 本音データとして DynamoDB に保存される
```

### 5.2 5分デモシナリオ（Q19=B 中心）

**「複数ツール文脈読解によるサボり提案」演出**

[1:00-2:00] 文脈読解のデモ
- 「提案資料の初稿作成」タスクをクリック

### 5.3 エッジケース・エラーシナリオ

| シナリオ | 対応 |

| **承認済みタスク** | ユーザーが「承認する」を押したタスク。サボり提案の対象になる |
| **Webhook 駆動** | 外部サービス（Slack 等）のイベント発生時に即座に処理を開始する方式 |
| **判断材料** | サボり判定の根拠として表示される箇条書きリスト（リマインド有無・メール温度感・会議日程等）|
