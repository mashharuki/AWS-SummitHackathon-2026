## 1. プロジェクト概要

### 1.1 プロダクトの本質

SABOROU は、AI 時代に増え続けるタスクに対して「今どうサボれば一番うまく逃げ切れるか」をリアルタイムに提案する AI エージェントサービスである。

タスクを早く終わらせるためのツールではなく、Slack・Gmail・Google Calendar といった外部ツール上の文脈（締切・依頼者の温度感・リマインド状況・関連会議）を読み取り、「なぜ今サボれるのか・どこから危ないのか」を具体的な根拠付きで提案する。

提供価値は2層に分かれる：

| 層 | 内容 |
|----|------|
| **表向きの価値** | タスクに追われる人の心に余白を作る |
| **裏設定の価値（"人をダメにする"）** | タスク整理能力・優先順位判断・危機管理能力・締切感覚を AI に委ねることで、その能力が自然に退化していく |

### 1.1.0 SABOROU の二重構造（コアデザイン哲学）

SABOROU は「支援ツール」と「人をダメにするツール」を同時に実現する。矛盾ではなく、意図した二重設計である。

| 役割 | 作用 |
|------|------|
| **表の顔（ユーザーが認識する価値）** | 科学的根拠に基づいて「今サボれる理由」を提示してくれる便利なツール |
| **裏の顔（人をダメにするメカニズム）** | AIが毎回「今サボっていい」と根拠付きで許可し続けることで、ユーザー自身の判断力が外部化・AI依存化していく |

**なぜ矛盾しないのか**:
- 科学的根拠（§1.1.2 の5理論）はAIの提案に「説得力」を持たせるための設計
- 説得力のある提案を繰り返し受けることで、ユーザーは「自分で判断しなくていい」という習慣が形成される
- これが §1.1.1 で示す「ダメになる4能力」の退化メカニズムの本質
- 結果として「AIに許可されないとサボれない」状態に陥る = 人をダメにするサービスの完成

### 1.1.1 ダメになる能力の具体例（裏設定）

| 能力 | 使用前（人間の状態） | 3ヶ月使用後（退化した状態） |
|------|-------------------|------------------------|
| **タスク整理能力** | メモ・メール・Slackを見て自分でタスクをリスト化 | AIが自動抽出するのを待つだけになり、メッセージを読んでも「何がタスクか」を自力で判断しづらくなる |
| **優先順位判断** | 締切・重要度・依頼者を考慮して優先順位を決定 | サボローの判定への依存が進み、「今やるべきか」を自力で決める機会が減る |
| **危機管理能力** | リマインドがなくても「そろそろ危ない」を察知できる | サボローの警告待ちになり、締切直前まで気づきにくくなる |
| **締切感覚** | 「あと2日」という時間感覚を体感で把握 | 「まだ大丈夫」の根拠を外部判断に依存し、内在化された時間感覚が弱まる |

### 1.1.2 サボり判定の科学的根拠（社会心理学・動機づけ理論）

SABOROU の「なぜ今サボれるか」という判定は、5つの確立された学術理論を設計基盤としている。
これにより、AIの恣意的な判断ではなく、人間行動の科学的メカニズムに基づく合理的提案であることを担保する。

| # | 理論 | 出典（査読済み） | 判定への活用 |
|---|------|----------------|-------------|
| 1 | **Collective Effort Model (CEM)** | Karau & Williams (1993). *Journal of Personality and Social Psychology, 65*(4), 681–706. | 「自分の貢献が価値ある結果につながる」と信じられないとき努力は合理的に低下する。締切が不明瞭・依頼者が認識していないタスクでは、今努力する価値が乏しいと判定 |
| 2 | **Identifiability（識別可能性）** | Williams, Harkins & Latané (1981). *Journal of Personality and Social Psychology, 40*(2), 303–311. | 個人の貢献が依頼者から「識別可能」な状態では社会的手抜きが消える。依頼者が offline / away でリマインドがない＝識別不能＝サボれる根拠 |
| 3 | **Sucker Effect（カモ効果）** | Kerr (1983). *Journal of Personality and Social Psychology, 45*(4), 819–828. | 依頼者自身が動いていないとき、「自分だけ先に動いて損をする」のを避ける心理が合理的に働く。依頼者の Slack ステータスや返信速度を参照 |
| 4 | **Self-Determination Theory (SDT)** | Ryan & Deci (2000). *American Psychologist, 55*(1), 68–78. | 外発的動機（締切プレッシャー・リマインド）がないとき、先延ばしが自律的に最適な戦略になる。リマインド回数・緊急度ラベルを定量化してLLMへ入力 |
| 5 | **Expectancy Theory** | Vroom (1964). *Work and Motivation.* Wiley. | 動機づけ＝期待（努力→成果）× 道具性（成果→報酬）× 誘意性。締切が遠い・フィードバックがないタスクは「今努力しても報われない」期待を生み、先延ばしが最適戦略となる |

**サボれる条件の論理式（5理論の統合）**:
```
[サボれる条件] = Identifiability が低い（依頼者が見ていない）
              ∧ Sucker Effect が発動（依頼者も動いていない）
              ∧ Expectancy が高い（締切まで余裕がある）
              ∧ SDT 外発的プレッシャーが低い（リマインドなし）
              → CEM によると集合的努力への動機が低下 = サボりが最適解
```

この科学的根拠に基づき、Claude Sonnet（Amazon Bedrock）がSlack・Gmail・Google Calendarから収集した文脈を評価し、「今サボれる理由」を人間が共感できる言語で説明する。

> 詳細な実装マッピングは [`application-design/component-methods.md`](../application-design/component-methods.md) §2「AG-02: SaboriProposerAgent」を参照。

## 3. 機能要件（FR）

### FR-01: 外部サービス連携・タスク自動抽出

**優先度**: MUST  
**根拠**: Q1=E（Slack 第1優先）、Q4=B（Webhook駆動）  
**v1.2.0 変更**: 連携サービスを Slack のみに絞り込み（Gmail / Google Calendar は v1.1.0 以降）

| 項目 | 詳細 |
|------|------|
| **連携サービス** | **Slack のみ**（v1.0.0 MVP スコープ）。Gmail / Google Calendar は v1.1.0 以降 |
| **トリガー方式** | Slack Webhook（Events API）/ リアルタイムイベント駆動（メッセージ受信時に即時処理） |
| **実装SDK（Slack）** | `@slack/bolt`（Node.js 版 Slack SDK）を採用し、Slack Event 受信・署名検証・メッセージパースを型安全に実装 |
| **実装SDK（チャットUI）** | Vercel AI SDK（npm: `ai`）の `useChat` フックを採用し、フロントエンドのサボローチャット UI をストリーミング表示 |
| **処理内容** | Slack メッセージを構造化タスク候補に変換（タスク名・締切・関係者・作業内容） |
| **エージェント** | エージェント①（タスク抽出エージェント）: Lambda + **Bedrock converse API + Tool Use**（AgentCore 不使用） |

**受入基準**:
- Slack のメッセージ受信からタスク候補が生成される（ウォームアップ済みLambdaで10秒以内、コールドスタート時は最大15秒）
- タスク候補に「タスク名 / 締切 / 依頼者 / 作業内容」が構造化されて格納される
- 生成されたタスク候補が DynamoDB「タスク候補」テーブルに書き込まれる

**v1.1.0 以降（スコープ外）**:
- Gmail の受信メールからのタスク候補生成
- Google Calendar の新規イベントからのタスク候補生成

### FR-01b: タスク構造化・手順整理（タスク整理AI）

**優先度**: SHOULD（v1.0.0 スコープ外 / v1.1.0 以降で実装）  
**根拠**: チーム追加要件（要素2）— エージェントパイプライン拡張（3エージェント構成）。M2 MVP（v1.0.0）では TaskExtractor と SaboriProposer の2エージェント構成で動作し、M3 決勝（v1.1.0）でタスク整理AIを追加する。

| 項目 | 詳細 |
|------|------|
| **エージェント** | エージェント①b（タスク整理エージェント）: U-03c: task-organizer / Lambda + Bedrock converse API + Tool Use |
| **入力** | TaskExtractorAgent（U-03a）が収集した生タスクリスト（複数タスク）|
| **処理** | タスク間の依存関係分析・実行手順の最適化・優先順位付け・「最もサボれる順序」の計算 |
| **出力** | 構造化されたタスクプラン（依存関係グラフ・推奨実行順序・サボり余地スコア）|
| **後続** | 整理済みタスク構造を SaboriProposerAgent（U-03b）に渡す |

**タスク整理AIが行う分析項目**:
- 各タスクの依存関係（タスクAが完了しないとタスクBが始められない等）
- 締切・重要度を考慮した優先順位マトリクス
- 「どの順番で実行すれば最も長くサボれるか」の最適パス計算
- 並行実行可能なタスクの特定（どれかを「寝かせ」ながら別件を進められる状況の検出）

**受入基準**:
- TaskExtractorAgent の出力を受け取り、30秒以内にタスク整理結果を返す
- タスク整理結果に「依存関係」「推奨実行順序」「サボり余地スコア（0〜100）」が含まれる
- 整理結果が DynamoDB「タスク整理ログ」テーブルに保存される
- SaboriProposerAgent がこの整理結果を参照してより精度の高い提案を生成できる

### FR-04: サボり提案のリアルタイム更新

## 4. 非機能要件（NFR）

### NFR-01a: パフォーマンス — タスク抽出レイテンシ（v1.0.0）

**根拠**: Q10=B  
**v1.2.0 変更**: Lambda Cold Start + Bedrock 推論の現実的レイテンシを反映した目標値に更新

- **目標**: Webhook 受信 → タスク候補生成（DynamoDB 書き込み完了）まで **ウォームアップ済みLambdaで10秒以内、コールドスタート時は最大15秒**
- **計測開始**: Slack Webhook POST 受信
- **計測終了**: DynamoDB TaskCandidates テーブルへの PutItem 完了
- **レイテンシ内訳（最悪ケース想定）**:
  - Lambda コールドスタート: 1〜3秒（Node.js + Hono）
  - Bedrock converse API 推論: 3〜8秒（Claude Sonnet、Tool Use 込み）
  - DynamoDB 書き込み: 0.1〜0.5秒
  - 合計: 最悪ケース 11〜12秒（ウォームアップ済みなら5〜9秒）
- **対象エージェント**: TaskExtractorAgent のみ（TaskOrganizerAgent は非同期・別計測）
- **Cold Start 対策**: デモ直前に Lambda をウォームアップするスクリプトを用意（Provisioned Concurrency は必要に応じて検討）
- **超過時のフォールバック**: CloudWatch Alarm + Lambda タイムアウト設定（60秒）で制御
- **UX 対策**: Webhook 受信後、フロントエンドに「処理中」バナーを表示し体感待ち時間を軽減

- **目標**: タスク承認 → サボり提案の初回ストリーミング表示（first chunk）まで **20秒以内**
- **計測開始**: ユーザーがタスク「承認する」ボタンを押す

- **ローディング状態**: 推論中はスケルトン / ローディングアニメーションで待機状態を明示

## 5. ユーザーシナリオ

### 5.1 主要ユーザージャーニー

#### ジャーニー1: 新規タスク候補の確認と承認

```
1. ユーザーが Slack でクライアントから「明日までに資料ざっくり作っておいて！」と受信
2. SABOROU の Webhook が Slack イベントを受信（Webhook トリガー）
3. エージェント①がメッセージを解析し、タスク候補を生成（10秒以内）
   → タスク名: 「提案資料の初稿作成」
   → 締切: 明日
   → 依頼者: クライアント（Slack DM）
4. タスク一覧画面の「承認待ちタスク」に新しいカードが表示
5. ユーザーが「承認する」ボタンをクリック
6. タスクが「承認済みタスク」セクションへ移動
7. エージェント②がサボり提案を生成開始
```

#### ジャーニー2: タスク詳細でサボり提案を確認

```
1. ユーザーが「承認済みタスク」のカードをクリック（タスク詳細画面に遷移）
2. 左ペインにタスクの前提情報（締切・やること・依頼者）が表示される
3. エージェント②がオンデマンドで文脈を再評価（10〜20秒以内）
4. 「現在のサボり判定」に1行の判定文が表示される
5. 「判断材料」に根拠の箇条書きが表示される
6. 右ペインのチャットでサボローが「まだ寝かせてOKだよぉ〜 ☁️」と提案
7. ユーザーがクイック返信「たしかに、まだ寝かせる」をクリック
8. 本音データとして DynamoDB に保存される
```

```
        ↓ DynamoDB「承認済みタスク」テーブル
        ↓
[エージェント②: サボり提案エージェント]
  - 実行環境: Lambda + Bedrock converse API + Tool Use（SSEストリーミング対応）
  - 入力: 承認済みタスク + Slack 文脈（温度感・スレッド）
  - 処理: 文脈統合 → サボり提案生成 → persona レンダリング
  - インタフェース: IBedrockClient（将来の AgentCore 移行に対応）
  - Persona: saboru_ottori（おっとりサボロー）
  - 出力: DynamoDB「提案ログ」テーブル → UI 表示（Lambda Response Streaming）
  
v1.1.0 以降（スコープ外）:
  - Gmail / Google Calendar コンテキスト収集
  - TaskOrganizerAgent（エージェント①b）
```

### 6.3 Persona 設計

- **MVP は単一人格固定**: `saboru_ottori`（おっとりサボロー）
- **判断と表現の分離**: エージェント②は内部的に verdict / reasoning / next_check_at を決定し、persona テンプレで表現（口調・絵文字）を整形
- **persona のデータ外出し**: DynamoDB `personas` テーブルに `persona_id / name / prompt_template / tone` を格納。将来の複数人格化に備えた構造
- **口調仕様**: 語尾「〜だよ」「〜かもぉ」、絵文字あり（☁️ 💤 😌 等）

## 9. 将来展望ロードマップ（MVP スコープ外）

本セクションは MVP（予選・決勝）のスコープ外であるが、SABOROU の長期ビジョンを示すために要件定義に含める。書類審査においてもビジョンの明確さは評価ポイントである。

### 9.1 展望1: サボりAI プロンプトの A/B テスト組み込み

**概要**: 人格A（おっとり共感系）と人格B（熱血反骨系）のどちらの提案がユーザーに刺さるかを A/B テストで計測し、Bedrock のプロンプト設計に反映する。

| 項目 | 内容 |
|------|------|
| **計測指標** | ユーザーエンゲージメント（サボり提案の承認率・「サボった」報告率・クイック返信率）|
| **A/B 割り当て** | ユーザー単位でランダム割り当て（実験グループを固定し再現性を担保）|
| **実装方針** | Bedrock のプロンプト設計に A/B バリアント ID を組み込む。DynamoDB の Personas テーブルに人格定義と実験フラグを格納 |
| **勝者判定** | 統計的有意差（p < 0.05）が出たタイミングで勝者人格を全ユーザーへ展開 |
| **将来拡張** | 勝者が決まった後も、時間帯・ユーザー特性ごとの「文脈依存人格選択」へ進化させる |

**期待効果**:
- 「人をダメにする」の効率が最大化される提案人格を科学的に特定
- ユーザーが最もサボりやすくなる口調・語り口のデータが蓄積される

### 9.2 展望2: 1対N 構造「サボり文化共有プラットフォーム」

**概要**: 現在の1ユーザー対1エージェントの1対1構造から、「サボり文化をコミュニティで共有・強化」する1対Nプラットフォームへ進化させる。

| 機能 | 内容 |
|------|------|
| **エコーチェンバー機能** | 「同じ状況でサボった人の体験談」を集積・表示。「みんなもサボってるから大丈夫」という社会的証明を提供し、サボりへの心理的ハードルを集団で下げる |
| **ランキング機能** | 「今週のサボり王」「最もクリエイティブなサボり提案」「最長連続サボり記録」などのランキング。ゲーミフィケーションでサボり文化を加速させる |
| **サボり体験談シェア** | 「このタスクをサボった結果○○になった」という体験談をコミュニティ共有。成功体験がエコーチェンバーとして循環する |
| **社会的比較機能** | 「あなたと似た状況（副業・フリーランス・3案件掛け持ち）の人は、今こんな感じでサボっています」という比較情報を表示 |

**「人をダメにする」の進化方向**:
- 個人レベル: AI に依存することで判断能力が退化する（現MVP）
- 集団レベル: コミュニティ全体でサボりが正当化・強化され、社会全体の生産性が低下する方向へ文化が醸成される（将来展望）

**技術展望**:
- マルチテナント DynamoDB 設計への移行
- リアルタイムランキング更新（ElastiCache / DynamoDB Streams）
- ユーザー間の匿名化済み体験談シェア機能

## 10. 制約事項・前提

### 10.1 AWS 制約（.claude/rules/aws-constraints.md 準拠）

- **リージョン**: ap-northeast-1（東京）のみ

- AI-DLC ワークフローに全開発作業が準拠
- 仕様書が先、コードが後（Spec 駆動開発）

- TypeScript 統一（フロント・バック・インフラ）

### 10.4 Bedrock 利用制約

- 1リクエストあたり最大 8,000 トークン（Q13=B）

- Bedrock 一部モデルは ap-northeast-1 以外のクロスリージョン推論が必要な場合あり（注意）
